# Docker Use

Docker is a powerful tool for creating, deploying, and running applications in containers. It allows you to package your application with all of its dependencies into a standardized unit for software development. This can be particularly useful for ROS 2 development, as it can help ensure that your development environment is consistent across different machines and operating systems.

> [!WARNING]
> It is prerequisite to have an understanding of Linux systems before using Docker.

> [!NOTE]
> This document assumes you have already set up your environment variables as described in the [Environment Variables documentation](./ENV.md) and that you have installed Docker. If you have not done so, please refer to that documentation before proceeding.

---

## Images

Docker images are the basis of Docker containers. They are read-only templates that contain the instructions for creating a Docker container. You can create your own Docker images or use pre-built images from Docker Hub. For ROS 2 development, you can use the official ROS 2 images provided by Open Robotics or for NVIDIA Jetson devices, you can use the images provided by NVIDIA. These images come pre-configured with ROS 2 and the necessary dependencies, making it easier to get started with development. We have used our own custom image based on the official ROS 2 image, which includes additional tools and libraries that we find useful for our development process. You can find our custom Dockerfile when we release our [Loon-Env](https://github.com/HumberASV/Loon-Env) repository.

> [!NOTE]
> For our Nvidia Orin Nano, we are using nvcr.io/nvidia/isaac/ros:aarch64-ros2_humble_4c0c55dddd2bbcc3e8d5f9753bee634c image, which is based on the official ROS 2 Humble image and includes additional tools and libraries for Loon-E's development on NVIDIA Jetson devices.

## Volumes

Docker volumes are a way to persist data generated by and used by Docker containers. They allow you to store data outside of the container's filesystem, which can be useful for sharing data between the host machine and the container or for persisting data across container restarts. When using Docker for ROS 2 development, you can use volumes to share your ROS 2 workspace between the host machine and the container. This allows you to edit your code on the host machine and have those changes reflected in the container without needing to rebuild the image.

Because Loon-E uses Zedx we have to make volumes to ensure the Zedx SDK and its dependencies are properly shared between the host machine and the container. This is necessary because the Zedx SDK requires access to certain hardware resources, such as the GPU, which may not be available within the container. By using volumes, we can ensure that the necessary files and libraries are accessible to the container, allowing us to use the Zedx SDK for vision processing in our ROS 2 development.

Following the example on [Stereolabs](https://www.stereolabs.com/docs/cameras/zed-x/zed-x-and-docker), we need to set the following volumes.

| Volume Name | Mount Path | Purpose |
| ------------- | ------------ |--------- |
| /dev/ | /dev/ | This is the device file for the Zedx camera. It allows the container to access the camera hardware. |
| /tmp/ | /tmp/ | This is a temporary directory that the Zedx SDK uses for storing temporary files. It allows the container to access these files as needed. |
| /var/nvidia/nvcam/settings/ | /var/nvidia/nvcam/settings/ | This is the directory where the Zedx SDK stores its configuration files. It allows the container to access these files as needed. |
| /etc/systemd/system/zed_x_daemon.service | /etc/systemd/system/zed_x_daemon.service | This is the systemd service file for the Zedx daemon. It allows the container to manage the Zedx daemon as needed. |
| ${HOME}/zed_docker_ai/ | /usr/local/zed_docker_ai/ | This is a directory on the host machine that we use for storing any additional files or data related to our ROS 2 development with the Zedx SDK. It allows us to easily share these files between the host machine and the container. |
| ${HOME}/.Xauthority | /root/.Xauthority | This is the X authority file for the host machine. It allows the container to access the X server for running graphical applications. |

## Options

When running a Docker container for ROS 2 development, there are several options that you may need to set to ensure that the container has the necessary permissions and access to resources. These options can include things like runtime settings, network settings, and GPU access. Setting these options correctly is important for ensuring that your ROS 2 development environment within the container functions properly and can communicate with other devices and services as needed.

| Option Name | Purpose |
| -------------- | ---------- |
| --runtime nvidia | This option allows the container to use the NVIDIA runtime, which is necessary for accessing the GPU resources required by the Zedx SDK. |
| -it | This option allows you to interact with the container's terminal, which can be useful for running commands and debugging within the container. |
| --privileged | This option gives the container additional privileges, which may be necessary for certain operations, such as accessing hardware resources or managing system services. |
| --net=host | This option allows the container to use the host machine's network stack, which can be useful for communication between the container and other devices on the network, such as the Zedx camera. |
| --ipc=host | This option allows the container to use the host machine's inter-process communication (IPC) namespace, which can be necessary for certain applications that require shared memory or other IPC mechanisms. |
| --gpus all | This option allows the container to access all available GPU resources on the host machine, which can be necessary for running applications that require GPU acceleration, such as those using the Zedx SDK for vision processing. |
| --pid=host | This option allows the container to use the host machine's process ID namespace, which can be useful for certain applications that require access to the host's process information or for debugging purposes. |

## Environment Variables

When running a Docker container for ROS 2 development, you may need to set certain environment variables to ensure that the container has the necessary configuration and access to resources. These environment variables can include things like display settings for graphical applications, ROS domain ID for communication, and RMW implementation for ROS 2 middleware. Setting these environment variables correctly is important for ensuring that your ROS 2 development environment within the container functions properly and can communicate with other devices and services as needed.

| Environment Variable | Purpose |
| ---------------------- | --------- |
| DISPLAY | This option allows the container to access the host machine's display, which can be useful for running graphical applications or visualizing data from the Zedx camera. |
| XAUTHORITY | This option allows the container to access the host machine's X authority file, which is necessary for running graphical applications that require access to the X server. |
| ROS_DOMAIN_ID | This option allows you to set the ROS domain ID for the container, which is necessary for ensuring that the container can communicate with other ROS nodes on the same network. |
| ROS_LOCALHOST_ONLY | This option allows you to limit ROS communication to localhost only, which can be useful for ensuring that the container's ROS nodes do not interfere with other ROS nodes on the network. |
| RMW_IMPLEMENTATION | This option allows you to specify which ROS middleware implementation to use within the container, which can be necessary for ensuring compatibility with the ROS nodes and libraries you are using in your development. |

## Example

Here is an example of how to run a Docker container for ROS 2 development with the necessary volumes, options, and environment variables:

```bash
docker run --runtime nvidia -it --privileged --net=host --ipc=host --gpus all --pid=host \
    -v /dev/:/dev/ \
    -v /tmp/:/tmp/ \
    -v /var/nvidia/nvcam/settings/:/var/nvidia/nvcam/settings/ \
    -v ${HOME}/zed_docker_ai/:/usr/local/zed_docker_ai/ \
    -v ${HOME}/.Xauthority:/root/.Xauthority \
    -e DISPLAY=$DISPLAY \
    -e XAUTHORITY=$XAUTHORITY \
    -e ROS_DOMAIN_ID=$ROS_DOMAIN_ID \
    -e ROS_LOCALHOST_ONLY=$ROS_LOCALHOST_ONLY \
    -e RMW_IMPLEMENTATION=$RMW_IMPLEMENTATION \
    --name loon-e-dev-container \
    loon-e-dev-image:latest
```

> [!NOTE]
> ROS_DOMAIN_ID, ROS_LOCALHOST_ONLY, and RMW_IMPLEMENTATION should be set in your host environment before running the container, and they will be passed into the container as environment variables. This ensures that the ROS 2 configuration within the container matches that of the host environment, allowing for proper communication and functionality of ROS 2 nodes and services.
